# Deck 图片可访问性 · 根因治理方案

> 目的：让 deck 里的图片在中国大陆稳定显示。本文档梳理问题根因、已完成的止血措施、
> 以及"根治方案"的分阶段实施计划、风险与决策点。供评估后分批实施。
>
> 状态：规划中（本文档为设计稿，尚未开始实施根因方案）。最后更新：2026-07-08。

---

## 1. 问题背景

deck 的每张图 = 一个存在 `slide.content`（jsonb）里的 `imageUrl` 字符串。渲染时
`<img src={imageUrl}>` 由浏览器去该地址下载。

**根因**：deck 生成时，文本大模型（`modules/deck/generation.service.ts`）直接把
`raw.githubusercontent.com/...` 之类的**外链**写进 `imageUrl`（`imageUrl` 只是个
`z.string().url()` 字段）。全流程**没有"下载转存到自有存储"的逻辑**。而
`raw.githubusercontent.com` 在中国被 DNS 污染 + 封锁，浏览器连不上 → 图裂开。

补充事实：全项目搜索 `raw.githubusercontent.com` **代码里一处都没有** → 这些地址
**是模型自己"编"出来的**（凭训练记忆猜了 `Eric-meiyan/pic-resource` 仓库路径），
因此还存在"地址不存在 / 图不对题"的隐患，不只是访问问题。

## 3. 根治目标

1. **访问**：新生成的 deck，图片托管到**自有 R2**，地址变成 `https://deckgene.com/assets/<key>`
   —— 和站点同域、同一套 Cloudflare，可访问性 = 站点本身，无第三方薄弱环节。
2. **来源**：让模型别再凭空编图片 URL，从源头减少"死链 / 不对题的图"。

已具备的基础设施：

- `src/routes/assets/$.ts` —— Worker 从 R2 读对象，公开、带 1 年 immutable 缓存。
- `src/routes/api/assets.ts` —— 上传原始字节到 R2，返回 `/assets/<key>`。
- `src/modules/deck/job.service.ts` + `job` 表 —— 现成的异步任务载体。
- `scripts/rehost-slide-images.mjs` —— 单页转存脚本（可复用其下载/上传/改写逻辑，也用于存量迁移）。

## 4. 核心设计原则（风险控制的总纲）

> **转存必须"旁路、异步、失败降级"，绝不阻塞生成、绝不影响积分、绝不删兜底。**

- 生成流程**照常先返回结果**（图先走 jsDelivr 兜底显示）。
- 转存放到**异步任务**里跑，成功了把 `imageUrl` 升级为 R2 地址，失败了保留原状（退回今天的状态，不会更糟）。
- 任何转存异常**都不影响生成成败、不触碰积分逻辑**。

按此原则，本方案的风险从"高"（动核心生成路径）压到"中低"。

## 5. 分阶段实施计划

### 阶段一：异步转存（新 deck）· 保留兜底 · 不动 prompt

**目标**：新生成的 deck，图片后台自动转存到 R2。风险最低、收益最直接。

- 生成完成后，派发一个**转存 job**（复用 `job` 表 / 或 Cloudflare Workflow）。
- Job 逻辑（复用 `rehost-slide-images.mjs` 的核心）：遍历 slide 的外链图 → 下载（走 jsDelivr 更稳）→ 传 R2（key 形如 `deck/<deckId>/<slideId>-<idx>.<ext>`）→ 把 `imageUrl` 改成 `https://deckgene.com/assets/<key>` → 写回库。
- **失败降级**：单张失败就跳过、保留原 URL；job 整体失败不影响 deck。
- **内容类型白名单**：只收 `image/png|jpeg|webp`；**拒绝 SVG**（防止在自有域名下托管可执行内容 → XSS）。
- **保留** `image-proxy.ts` 的 jsDelivr 改写作为兜底。

**验收**：新生成 deck 一段时间后，图片地址变为 `deckgene.com/assets/...` 且中国可访问；生成本身耗时、成功率、积分不受影响。

### 阶段二：源头治理（prompt）· 最小改动先行

**目标**：减少模型凭空编造图片外链。

- **最小改动**：在生成 prompt 里加一条——"拿不准就把 `imageUrl` 留空，不要编造图片地址"。
- 观察对 deck 视觉完整度的影响（可能变空 → 权衡产品体验）。
- **（可选，重）** 进一步：让模型只输出"图片描述/搜索词"，由受控步骤 AI 生图或搜正版图再存 R2 —— 这是**新增子系统**（成本/延迟/新失败点），单独评估，不在本阶段强推。

**验收**：新 deck 里 `raw.githubusercontent` 外链显著减少；deck 视觉质量无明显倒退。

### 阶段三：收尾与治理

- **删除清理**：`deck.service.ts` 的 `deleteDeck` 里，顺带删该 deck 的 R2 图，避免孤儿文件。
- **存量迁移（单列、单独灰度）**：用脚本把**已有 deck** 的 GitHub 图批量转存 R2。这是**触碰全部生产数据**的操作，风险更高，务必先干跑、分批、可回滚。
- **（可选）压缩**：转存时把大图压到几百 KB（异步环境可用图片处理），进一步省存储。

## 6. 风险清单与缓解

| 级别 | 风险                                                                          | 缓解                                                  |
| ---- | ----------------------------------------------------------------------------- | ----------------------------------------------------- |
| 🔴   | 动生成主流程，bug 会让**所有 deck 生不出来**（爆炸半径大）                    | 转存**旁路异步 + 失败降级**；灰度上线 + 一键回滚      |
| 🔴   | Worker 限制：同步下载多图会**拖慢生成 / 撞超时 / 触子请求·内存上限**          | **不在生成内同步做**，改异步 job/Workflow             |
| 🔴   | 积分一致性：转存失败若牵连生成失败 → 退积分逻辑复杂                           | 转存**绝不影响生成成败**，积分逻辑不动                |
| 🟡   | 改 prompt 是行为改动、LLM 牵一发动全身；可能不听话 / deck 变空 / 波及输出质量 | prompt **最小改动**先行；充分回归；生图子系统单独评估 |
| 🟡   | 在自有域名托管模型指定的任意图，**SVG 可带脚本 → XSS**                        | 转存**白名单 content-type**，拒绝/消毒 SVG            |
| 🟡   | R2 **只增不减**、删 deck 不清图 → 孤儿文件堆积                                | 删除时清理 + 定期扫孤儿                               |
| 🟢   | 本地无 R2 绑定（走 public/uploads 兜底），**本地测不出线上行为**              | 预发/灰度环境验证                                     |
| 🟢   | 下载源失败：GitHub 429 限流 / 死链 404                                        | 走 jsDelivr 下载、有限重试、失败降级                  |

## 7. 存储 / 成本评估（结论：不是瓶颈）

- **出站流量在 R2 完全免费** —— 服务图片给中国用户，带宽成本 0（相对 S3 的杀手锏）。
- **存储 $0.015/GB·月，前 10GB 免费**：压缩后约 3000 个 deck / 不压缩约 800 个 deck 才用满 10GB；100GB = $1.5/月，1TB = $15/月。
- **读取**：`/assets/$` 设了 1 年 immutable 缓存，重复访问命中边缘缓存、不碰 R2，读取次数极低。
- **真正要管的不是容量，是"只增不减"的堆积** → 见阶段三"删除清理"。
